# Algorithm Integration API

## Purpose

This document is the handoff contract for audio-algorithm work. MineVOICE keeps Minecraft events, UDP relay, device capture, and DSP separate. Algorithm implementations should depend on `common` only.

The current product routes audio first. It sends only authorized frames to a listener:

- `PROXIMITY`: sender and listener must be in the same dimension and within the standalone server `proximityDistance`.
- `GROUP`: sender and listener must belong to the same MineVOICE group; distance is ignored.

The relay does not decode, mix, or trust a client supplied position. Minecraft server state is signed with the server `sharedSecret` and refreshed once per second.

## Stable Java Contract

Algorithm code should implement:

```java
public interface VoiceFrameProcessor {
    VoiceProcessingResult process(VoiceProcessingRequest request);
}
```

The processor receives:

```java
public record VoiceProcessingRequest(
    VoiceFrame frame,
    VoiceEnvironmentContext environment
) {}

public record VoiceEnvironmentContext(
    VoicePlayerState sender,
    VoicePlayerState listener,
    VoiceChannel channel,
    List<VoiceBlockSample> blockSamples
) {}
```

The processor returns a listener-specific result:

```java
public record VoiceProcessingResult(
    byte[] encodedAudio,
    float gain, // [0, +infinity)
    float pan   // [-1, 1]
) {}
```

Do not mutate `VoiceFrame.encodedAudio()` or any input list. Return a new byte array when audio content changes.

## Authoritative Inputs

`VoicePlayerState` is generated by the Minecraft server, not by a client:

| Field | Meaning |
| --- | --- |
| `playerId` | Minecraft player UUID |
| `playerName` | Current game profile name |
| `dimensionId` | Resource location, for example `minecraft:overworld` |
| `x`, `y`, `z` | Server-side player position |
| `groupId`, `groupName` | Current ephemeral MineVOICE group, or `null` / empty |
| `muted` | Microphone state announced by that client through the Minecraft connection |

`VoiceBlockSample` is the deliberately small block input for occlusion work:

| Field | Meaning |
| --- | --- |
| `x`, `y`, `z` | Block coordinates |
| `blockStateId` | Namespaced block-state identifier |
| `occluding` | Server policy says the block participates in line-of-sight attenuation |
| `absorption` | Normalized material absorption in `[0, 1]` |

Use `VoiceEnvironmentProvider` in the Minecraft-server integration to gather only the samples required by an algorithm. The first implementation should sample a bounded line or a small capsule between the sender and listener. Do not scan an entire sphere for every 20 ms frame.

## Transport Design

Audio and world context travel on different cadences:

1. `VOICE_FRAME` carries latency-sensitive encoded audio and its `VoiceChannel`.
2. `SERVER_STATE` carries HMAC-authenticated positions, dimensions, groups, and mute state once per second and immediately after group/status changes.
3. A future `VoiceEnvironmentProvider` produces the bounded block samples for the already-authorized sender/listener pair.

This is intentional. Repeating blocks in every audio datagram would waste bandwidth and make packet loss affect both voice and world context. The algorithm sees the two inputs together through `VoiceProcessingRequest`.

## Processor Rules

- Never expand the recipient set. Routing remains in `VoiceRelayService`.
- Treat group members outside normal voice distance as valid only when `channel == GROUP`.
- Return `gain = 0` for a valid but fully occluded result; do not throw for normal occlusion.
- Clamp internal values before returning them. MineVOICE clamps `gain` and `pan` again at the boundary.
- Keep expensive caches keyed by dimension, chunk, and block revision. Invalidate on block changes rather than sampling all blocks every frame.
- Do not put Minecraft classes in `common`; adapter code belongs in `minecraft-neoforge`.

## Integration Sequence

1. Implement a `VoiceEnvironmentProvider` in `minecraft-neoforge` that extracts bounded block samples from `ServerLevel`.
2. Implement `VoiceFrameProcessor` in a separate algorithm module depending on `common`.
3. Add the processor at the per-listener playback boundary after relay authorization.
4. Start with `gain` and `pan`; add codec-aware transforms only after Opus replaces `MockVoiceCodec`.
5. Validate with two clients on opposite sides of a wall, then measure CPU cost with `client-sim` before increasing block radius.

## Current Limits

`VoiceEnvironmentProvider` and `VoiceFrameProcessor` are stable interfaces in this revision, but no production block sampler or server-side DSP processor is wired yet. The current client playback path performs basic proximity stereo: loaded client entities are mixed into 48 kHz stereo PCM with distance gain and listener-relative left/right panning. Group frames remain centered so remote group members do not disappear when their entity is not loaded. Block occlusion, material absorption, and codec-aware DSP remain separate follow-up work.
